```markdown
# allocate_markets_to_moms 設計メモ

## 1. 目的

`allocate_markets_to_moms(..)` は、**demand anchored lot を、どの MOM subtree に入れるか決める政策実行関数**である。

この関数は、Outbound 側で需要起点として見えている lot 群を読み取り、
lot が示す市場情報に基づいて、
Inbound 側のどの MOM が担当するかを決め、
各 MOM の demand layer に lot を配置する。

この関数で決まった配分結果が、その後の Inbound planning の起点になる。

---

## 2. この関数の責務

この関数の責務は、以下の 4 点に限定する。

1. source lots を選ぶ
2. lot_id から market key を抽出する
3. policy に基づいて primary MOM を決める
4. 選ばれた MOM の `psi4demand[w][0]` に lot を配置する

### 責務に含めないもの

この関数は以下を担当しない。

- MOM capacity 制約
- MOM leveling
- MOM から PAD への forward planning
- MOM から DAD への handoff
- Outbound tree 側の distribution planning
- KPI 集約
- root / supply_point の再集約

---

## 3. 入力

### 関数引数

- `out_root`
  - outbound tree の root node
- `in_root`
  - inbound tree の root node
- `policy: dict`
  - 市場キーごとの MOM 優先ルール
- `source_layer: str = "outbound_supply"`
  - source lots をどこから読むか
- `weeks: int | None = None`
  - 対象週数の上限
- `clear_existing_mom_demand: bool = True`
  - 再配賦前に MOM demand をクリアするか
- `debug: bool = True`
  - debug log を出力するか

### 実質入力データ

- `out_root.psi4supply[w][0]` または `in_root.psi4demand[w][0]`
- lot_id naming rule
- inbound tree 内の `MOM_` prefix node 一覧
- policy の市場 → MOM 対応表

---

## 4. 出力

### 返り値

- `return out_root, in_root`

### 実質出力

- `in_root` 配下の各 `MOM` node の `psi4demand[w][0]` が更新される
- debug 時に配賦結果 summary が出力される

この関数の本質は、返り値ではなく **MOM demand layer の破壊的更新**である。

---

## 5. 現在の処理手順

1. `MOM_` prefix の node を inbound tree から集める
2. `source_layer` に応じて source lots の基点を決める
3. 必要なら各 MOM の demand slot をクリアする
4. week ごとに source lots を走査する
5. lot_id から market key を抽出する
6. policy に基づいて primary MOM を選ぶ
7. 対応する MOM の `psi4demand[w][0]` に lot を append する
8. debug 用に MOM ごとの件数集計を出す

---

## 6. 前提条件

この関数が正しく動くためには、以下の前提が必要である。

### 前提 1: source lots は demand anchored である
現在は `out_root.psi4supply[w][0]` を source にする使い方が主であり、
そこで扱う lot は最終需要先を表す情報を持っていることが前提である。

### 前提 2: lot_id から市場キーが抽出できる
`_default_market_key_from_lot(..)` は naming rule に依存して market key を抽出する。

### 前提 3: MOM node は `MOM_` prefix で識別できる
node naming rule が関数動作の前提になっている。

### 前提 4: 1 lot は 1 primary MOM にしか配賦しない
現実装は split allocation を行わず、1 lot -> 1 MOM の写像である。

### 前提 5: capacity 制約は後段で扱う
MOM capacity 制約はこの関数では扱わず、
後段の leveling / capacity 処理で扱う。

---

## 7. 現在の設計上の強み

### 7.1 責務が細い
source selection / market extraction / policy execution / MOM write に責務が限定されている。

### 7.2 MOM subtree planning と相性が良い
この関数は MOM subtree の入口を作り、
その後の Inbound planning を MOM subtree 単位で閉じる設計と整合する。

### 7.3 Outbound 側への handoff 設計の雛形になる
将来の `MOM -> DAD` handoff においても、
「lot の demand anchor を見て subtree 入口へ振る」という同じ思想を再利用できる。

---

## 8. 現在の注意点

### 8.1 clear 対象が slot0 に偏っている
再配賦時に `psi4demand[w][0]` のみ clear している場合、
他 slot に stale data が残る可能性がある。

### 8.2 lot_id naming rule 依存が強い
market key 抽出が naming rule に依存するため、
命名変更時に壊れやすい。

### 8.3 allocation 結果が構造化されて返らない
現状は PSI を直接更新するのみであり、
後続で配賦結果を追跡しづらい。

### 8.4 capacity-aware ではない
この関数は policy による primary MOM 決定までであり、
capacity を見て secondary MOM に逃がすような logic は持っていない。

---

## 9. 将来拡張点

### 9.1 allocation map を返す
以下のような assignment 情報を返せると、後段処理が大幅にやりやすくなる。

- `lot_to_mom`
- `mom_to_lots`
- `week_mom_counts`

### 9.2 market key 抽出を naming rule 依存から分離する
将来的には、lot metadata や explicit demand anchor object から市場情報を引く方が望ましい。

### 9.3 policy 表現を強化する
今後の拡張候補:

- primary / secondary MOM
- split ratio
- region priority
- product family 別 policy
- period 別 policy

### 9.4 capacity-aware allocation への発展
基本方針としては capacity は後段責務とするが、
将来的には
「primary MOM で受ける -> overflow を secondary MOM に逃がす」
二段階 allocation への拡張余地がある。

### 9.5 DAD 側 handoff 関数への対称拡張
将来的には以下のような対称関数が必要になる。

- `allocate_lots_to_dads(..)`

これは、

- MOM で確定した supply lot を source にし
- lot_id から最終 leaf を見て
- leaf から DAD を特定し
- DAD subtree の入口へ lot を seed する

関数である。

---

## 10. WOM における位置づけ

`allocate_markets_to_moms(..)` は、
WOM の Inbound / Outbound 統合において、
**需要起点 lot を Inbound 製造担当 subtree に受け渡す policy execution layer**
として位置づけるのが適切である。

推奨する全体構成は以下。

1. Outbound 側で demand anchored lot を形成する
2. `allocate_markets_to_moms(..)` で MOM subtree 入口へ配賦する
3. Inbound 側は MOM subtree 単位で planning を行う
4. MOM で確定した supply lot を DAD へ handoff する
5. Outbound 側は DAD subtree 単位で planning を行う
6. 必要なら上位ノード KPI は後段で集約する

---

## 11. 設計上の原則

### 原則 1
`allocate_markets_to_moms(..)` は **policy executor** に徹する

### 原則 2
capacity は後段責務とする

### 原則 3
MOM subtree で Inbound planning を閉じる

### 原則 4
MOM -> DAD handoff は別の bridge layer として定義する

### 原則 5
上位集約表示は planning 本体と分離する

---

## 12. 一言でまとめると

`allocate_markets_to_moms(..)` は、

**「demand anchored lot を、どの MOM subtree に入れるか決める政策実行関数」**

である。

ここを正しく理解し、

- capacity は後段
- DAD handoff は次段
- KPI 集約はさらに後段

と責務分離することが、WOM の Inbound / Outbound 統合設計において最も見通しが良い。
```

---

ここから先の一般論ですが、はい、おっしゃる通りです。
**「実需要に対する capacity 付きの生産配分問題」**は、サプライチェーン需給バランスの中核問題の一つです。

そして、一般には **1つの万能手法で解く**というより、
**意思決定レイヤごとに違うモデルを組み合わせる**のが主流です。
生産計画の研究レビューでは、集約計画・階層計画・MRP・能力計画・サプライチェーン計画など複数レイヤがあり、手法も分析モデル、AI、シミュレーションなどに分かれます。 ([MyWeb][1])

## 代表的なアプローチ

**数理最適化**は、いまでも中核です。
典型的には LP/MILP、動的計画、確率計画、ロバスト最適化で、能力制約つき lot sizing や multi-stage production-inventory planning の形に落とします。利点は、制約と目的を明示でき、説明可能で、シナリオ比較に強いことです。弱点は、現実の複雑な運用ルールを全部きれいに入れるとモデルが重くなりやすいことです。 ([MyWeb][1])

**ヒューリスティック / 分解法**は、実務では非常に重要です。
lot sizing / scheduling の古典的研究でも、厳密解だけでなく近似法や分解が重要でしたし、現代の高複雑ネットワークでも実務側はしばしばヒューリスティック統合に頼っています。利点は速いこと、現場ルールを入れやすいことです。弱点は、最適性保証や一貫性が弱くなりやすいことです。 ([サイエンスダイレクト][2])

**機械学習**は、単独で「生産配分の意思決定」を担うというより、
**需要分布予測・能力予測・パラメータ推定**に強いです。
最近の MPS 研究でも、確率需要を ML で予測し、その出力を stochastic optimization に渡す構成が有効でした。利点は非線形・非定常パターンの学習です。弱点は、制約充足や説明可能性を ML 単独では担いにくいことです。 ([サイエンスダイレクト][3])

**シミュレーション / シミュレーション最適化**は、不確実性や運用ルールが強いときに有効です。
最近は simulation と ML のハイブリッドが増えていますが、レビューでは「最適化が十分組み込まれていない」ことも指摘されています。利点は現実忠実性と what-if に強いこと、弱点は探索が重くなりやすいことです。 ([サイエンスダイレクト][4])

**強化学習**は、逐次意思決定・高頻度な適応制御には有望です。
ただし、Production Planning and Control の文脈では、dispatching など一部タスクでは強い一方、広範な PPC 全般で industrial maturity が十分高いとはまだ言いにくく、良い simulator・状態設計・データ・実装体制が必要です。利点は動的適応、弱点は状態空間の大きさ、収束、現場実装の難しさです。 ([スプリンガーリンク][5])

## PROS / CONS を一言でまとめると

**MILP / 確率・ロバスト最適化**
最も説明しやすく、WOM と相性が良いです。
ただし、粒度を上げすぎると重くなります。 ([MyWeb][1])

**ヒューリスティック / 分解**
速くて実装しやすいです。
ただし、全体整合と最適性は弱くなりがちです。 ([サイエンスダイレクト][2])

**ML**
予測とパラメータ推定に強いです。
ただし、意思決定コアを単独で任せるのは危険です。 ([サイエンスダイレクト][3])

**RL**
動的制御には将来性があります。
ただし、現時点では「万能の best solution」ではありません。 ([スプリンガーリンク][5])

## best solution は何か

**単一手法の best solution はありません。**
実務上の best は、ほぼ確実に **ハイブリッド構成**です。

私のおすすめは、WOM と整合的に次の構成です。

**1. 予測層**
需要予測、需要分布推定、能力予測。
ここは ML を使ってよいです。 ([サイエンスダイレクト][3])

**2. 政策実行層**
`allocate_markets_to_moms(..)` のような、需要起点 lot を subtree 入口へ振り分ける層。
ここはルールベース + explainable な policy engine が向いています。 

**3. 計画最適化層**
能力制約つき lot sizing / production allocation / inventory positioning。
ここは MILP / stochastic / robust optimization が中核です。 ([MyWeb][1])

**4. 実行シミュレーション層**
WOM の event / PSI / flow simulation。
ここで現実のリードタイム、在庫、push/pull、例外を検証します。 ([サイエンスダイレクト][4])

**5. 適応制御層**
必要なら RL や bandit を入れる。
ただし、最初から中核に置くより、dispatching や policy tuning に限定した方が安全です。 ([スプリンガーリンク][5])

## WOM の上位モデルとしてのおすすめ構成

WOM の上位には、私は次のモジュール構成が最も良いと思います。

**Forecast / Sensing**
需要、供給、能力、リスクの予測。

**Policy Layer**
`allocate_markets_to_moms(..)` と、将来の `allocate_lots_to_dads(..)`。
ここが「誰が担当するか」を決めます。

**Optimization Core**
capacity つき production allocation、safety stock、inventory positioning、transport allocation。
ここが「どれだけやるか」を決めます。

**WOM Planning Engine**
PSI / event / push-pull / subtree planning。
ここが「どう流れるか」を作ります。

**Scenario & Simulation Layer**
what-if、地政学リスク、capacity disruption、demand shock。

**Decision Cockpit**
KPI、explanation、trade-off、policy comparison。

この構成だと、WOM は単なる可視化ツールではなく、
**政策実行と最適化をつなぐ意思決定 OS** のような位置づけになります。

一言でまとめると、
**WOM の best future は、ML が予測し、optimization が決め、WOM engine が流し、simulation が検証するハイブリッド構成**です。
その入口として、`allocate_markets_to_moms(..)` を正しく定義するのは、かなり本質的な一歩です。

次に進めるなら、
**`allocate_markets_to_moms(..)` と対になる `allocate_lots_to_dads(..)` の設計メモ** を作るのが一番自然です。
