## 1. Purpose

This memo defines the design for the **Explicit Pipeline Management Cockpit KPI View**.

The explicit pipeline now has a connected explanation stack inside the planning sequence:

```text
explicit bridge + capacity pipeline
    ↓
reporting flag switchboard helper
    ↓
capacity report
    ↓
issue candidates
    ↓
Cost / KPI enrichment
    ↓
exportable audit files
```

The next step is to define how these results should be presented to a manager or planner in a read-only Management Cockpit KPI view.

This memo is a design document only.

It does not implement GUI code.

## 3. Design Goal

The goal is to define a GUI view that shows the most important explicit pipeline reporting / issue / Cost-KPI results in a management-friendly way.

The view should answer:

```text
1. What happened?
2. What is the current planning / capacity risk?
3. Which issue candidates were generated?
4. Which issues have high business impact?
5. What Cost / KPI assumptions were used?
6. What files were exported?
7. What should a manager review next?
```

The view should not make decisions.

The view should not execute replan commands.

The view should not silently create files.

The view should present already-generated evidence.

---

## 4. Non-Goals

This design does not implement:

```text
new planning logic
new Cost / KPI calculation
new issue candidate mapping
new export logic
automatic replanning
ReplanCommand execution
OR optimization
database persistence
Knowledge Continuity persistence
approval workflow
formal accounting
```

This phase is only about:

```text
read-only GUI / Management Cockpit view design
```

---

## 5. Core Principle

The Management Cockpit KPI View should be:

```text
a viewer over explicit pipeline artifacts
```

not:

```text
a hidden execution engine
```

This distinction is critical.

The planning sequence and reporting stack produce objects.

The cockpit displays those objects.

The cockpit should not secretly trigger:

```text
planning execution
exports
Cost / KPI enrichment
replan commands
knowledge persistence
```

unless a future explicit user action and feature flag are designed.

## 8. Recommended View Entry Point

Potential method name in GUI:

```python
_show_explicit_pipeline_management_cockpit(self)
```

or:

```python
_open_explicit_pipeline_kpi_view(self)
```

Potential helper function:

```python
build_explicit_pipeline_kpi_view_model(env) -> dict
```

Recommended approach:

```text
1. Build a pure view model from env.
2. Test the view model without Tk.
3. Use Tk only to render the already-built view model.
```

This keeps tests stable and avoids brittle GUI tests.

## 10. Missing Data Behavior

The view should never crash merely because a reporting object is missing.

If there is no explicit pipeline result:

```text
available = False
message = "No explicit pipeline result is available. Run planning with explicit pipeline enabled."
```

If there is a pipeline result but no issue candidates:

```text
show capacity report if available
show "No issue candidates generated or issue candidate flag is off."
```

If there is issue candidate bundle but no Cost / KPI bundle:

```text
show issue candidates
show "Cost / KPI enrichment is not available or flag is off."
```

If there are no export results:

```text
show "No export result is available. Export flags may be off."
```

This keeps the view safe for partial configurations.

Recommended view model:

```python

    "available": bool,

        "explicit_pipeline_result": bool,

        "issue_candidates": bool,
        "cost_kpi_bundle": bool,
        "capacity_report_export": bool,
        "issue_candidate_export": bool,

    },
    "executive_kpi_summary": {
        "currency": str,
        "estimated_total_business_impact": float,
        "estimated_lost_sales_value_total": float,
        "estimated_margin_impact_total": float,
        "estimated_inventory_cost_impact_total": float,
        "estimated_capacity_cost_impact_total": float,
        "estimated_service_penalty_total": float,
        "impact_values_are_directional": bool,
        "double_counting_possible": bool,

    "capacity_summary": {...},
    "issue_summary": {...},
    "top_impact_issues": [dict],
```

## 23. Sorting and Filtering

Initial MVP should support deterministic default sorting only.

Recommended:

```text
top_impact_issues:
    estimated_total_business_impact descending
    severity priority
    issue_type
    node
    week

health issues:
    severity priority
    issue_type
    source

replan candidates:
    expected_benefit_category
    command_type
    node
    week
```

Do not add interactive filters in MVP unless very small.

A later GUI version can add:

```text
filter by severity
filter by issue_type
filter by node
filter by product
filter by impact_category
```

## 27. Empty State Behavior

If no explicit pipeline data exists:

```text
show message:
"No explicit pipeline reporting data is available. Run planning with explicit pipeline enabled."
```

If report exists but no Cost / KPI:

```text
show available report / issue sections
show message:
"Cost / KPI enrichment is not available. Enable the Cost / KPI flag and rerun planning."
```

If exports are missing:

```text
show message:
"Export results are not available. Export flags may be off."
```

No exception should be thrown.

---

## 28. Safety Rules

The GUI view must not:

```text
mutate planning data
execute replan commands
write export files
change feature flags
rerun planning
hide missing-data warnings
present directional estimates as formal accounting
```

The view may:

```text
read env objects
format summaries
sort rows
display paths
display warning messages
```

---

## 29. Cost / KPI Caveat Display

The cockpit must show a caveat when Cost / KPI data exists:

```text
Cost / KPI values are directional scenario estimates.
They are not formal accounting values.
Double counting may be possible depending on assumptions.
```

This aligns with the summary flags:

```text
impact_values_are_directional
double_counting_possible
```

This is important for management trust.

---

## 30. Relationship to Existing Management Cockpit

WOM already has Management Cockpit concepts and business report builder work.

This explicit pipeline KPI view should be positioned as:

```text
explicit pipeline diagnostic / issue / impact view
```

not as a replacement for all management reporting.

It should complement:

```text
business report
cost waterfall
allocation breakdown
scenario comparison
```

The explicit pipeline KPI view focuses on:

```text
capacity-risk-driven issue candidates and their directional business impact
```

## 31. Relationship to Knowledge Continuity

Future actions may include:

## 36. Summary

The explicit pipeline reporting / issue / Cost-KPI stack is now connected to the planning sequence.

The next step is presentation.

The Management Cockpit KPI View should be a read-only management-facing view over already-generated objects:

```text
capacity report
issue candidates
Cost / KPI enriched issue candidates
export results
assumptions
```

The central table should be:

```text
Top Business Impact Issues
```

because it converts operational planning trouble into management attention.

The guiding rule is:

```text
Display evidence.
Do not secretly execute.
Do not pretend directional estimates are accounting truth.
```

This keeps WOM trustworthy as it moves from planning engine toward management decision-support cockpit.
