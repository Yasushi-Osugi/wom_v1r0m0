# MOSD Phase 2B+1 Purchase Cost Propagation

## Purpose

Connect parent node `ship_price_per_lot` to child node `purchase_cost_per_lot` through product_tree edges.

## Background

Revenue calculation was already active, but `purchase_cost_per_lot` and `purchase_amount` stayed zero in `node_money_eval.csv`.

## Implemented behavior

- Explicit child `purchase_cost_per_lot` is respected when non-zero.
- If child `purchase_cost_per_lot` is missing or zero, parent `ship_price_per_lot` is propagated.
- Existing edge-level transfer/freight/insurance/duty/handling fallback remains active when available.
- `purchase_amount` is calculated using propagated `purchase_cost_per_lot` and the evaluator's current inbound lot basis.
- Revenue calculation remains based on `ship_price_per_lot`.
- Inventory valuation remains separate from P/L purchase cost.

## Not included

- Full node price formation
- Target costing / downward allowable cost propagation
- Management Cockpit KPI integration
- Inventory B/S to P/L bridge

## Safety requirements

Do not modify:

- existing GUI behavior
- existing planner behavior
- existing MOSD adapter behavior unless strictly necessary
- existing `data/*.csv` master files as committed fixtures

All runtime output should continue to be generated by existing output mechanisms.

## Acceptance criteria

After implementation, running the current WOM flow with:

- `supply_point ship_price_per_lot = 600`
- `DAD_US_CENTRAL_DC ship_price_per_lot = 700`
- `CS_US_ECOM ship_price_per_lot = 800`

should produce `node_money_eval.csv` where:

- `DAD_US_CENTRAL_DC.purchase_cost_per_lot == 600`
- `CS_US_ECOM.purchase_cost_per_lot == 700`
- `DAD_US_CENTRAL_DC.purchase_amount > 0`
- `CS_US_ECOM.purchase_amount > 0`

Revenue values should remain non-zero and based on each node's `ship_price_per_lot`.
Profit should no longer equal revenue at `CS_US_ECOM` when `purchase_cost_per_lot` is propagated.
